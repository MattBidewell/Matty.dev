# Interactive Blog Post System for matty.dev

Date: 2026-03-14
Branch: `feat/setup-interactive-essays`

This file captures the planning conversation for adding interactive essays to `matty.dev`, so the work can be resumed on another machine.

## Original Goal

Introduce a new category of interactive posts to `matty.dev`: long-form technical essays with inline animated and interactive demos that teach distributed systems and AI tooling fundamentals.

The desired qualities are:

- Inline demos that live inside the essay flow, not on separate pages and not in iframes.
- Interactive exploration over passive animation.
- Posts that still feel like articles first, not mini apps.
- Clean, minimal presentation.

## Inspiration

Reference sites:

- `https://shiptypes.com/`
- `https://benjdd.com/stylized-image-binning-algorithm/`

Specific qualities to emulate:

1. Concept -> interactive demo -> code -> next concept.
2. Step-through visual explanations.
3. Canvas/SVG-based teaching visuals embedded directly in the prose.
4. Focused writing and restrained UI.

## Candidate Topics

- Consistent hashing
- DAG resolution
- Edge caching strategies
- How a GraphQL query resolves
- Prompt engineering patterns

## Constraints And Preferences

- TypeScript everywhere.
- Performance matters; keep posts mostly static and lazy-load demos.
- Progressive enhancement; posts must remain readable without JavaScript.
- Keep the build pipeline simple and dependency count low.
- Accessibility matters: ARIA labels, keyboard support where sensible, reduced-motion support.
- Must be maintainable by one person.

## Current Site Assessment

The repo is already a good fit for this feature. No framework switch is needed.

### Existing Blog Pipeline

- Content lives in `_posts/*.md`.
- Frontmatter and body parsing are handled by `lib/api.tsx` using `gray-matter`.
- Post pages render Markdown with `Remarkable` in:
  - `app/blog/[slug]/page.tsx`
  - `app/mumblings/[slug]/page.tsx`
  - `app/projects/[slug]/page.tsx`
- The rendered HTML is injected via `dangerouslySetInnerHTML` in `app/components/blog/body/PostBody.tsx`.
- Static generation already exists through `generateStaticParams()`.

### Main Blocker

The current Markdown -> HTML string -> `dangerouslySetInnerHTML` pipeline prevents true inline React/TypeScript demos.

### Recommendation

Stay on Next.js App Router and static generation.

- Keep existing `.md` posts working.
- Add MDX support for interactive essays.
- Render prose on the server.
- Keep interactive demos as small client islands embedded inside MDX.

This is the smallest change that unlocks inline interactive content.

## Architecture Recommendation

### High-Level Direction

- Remain on the current Next.js stack.
- Add dual support for `.md` and `.mdx` posts.
- Use MDX only for posts that need inline interactive components.
- Preserve static generation and Netlify deployment.

### Why Not Switch Frameworks

Astro would also work well for this pattern, but moving the site is unnecessary overhead. The current app already has the right shape; only the content rendering layer needs to evolve.

### Recommended Content Model

Keep legacy posts in `_posts/*.md`.

Add a folder-based content structure for interactive essays so prose, demos, and assets are co-located:

```text
content/
  posts/
    consistent-hashing/
      index.mdx
      meta.ts
      demos/
        RingIntroDemo.tsx
        RebalanceDemo.tsx
        VirtualNodesDemo.tsx
      assets/
        ring-fallback.webp
```

Benefits:

- Post-specific demo code stays with the post.
- Assets and still-image fallbacks live beside the prose.
- No giant global demo directory.

## Library Recommendations

### Use

- `@next/mdx` for interactive essay rendering.
- `rehype-highlight` or the existing `highlight.js` approach for code blocks.
- Native SVG for most diagrams and educational visualizations.
- Small utility hooks/components for playback and responsiveness.

### Optional Later

- `motion` or Framer Motion for subtle UI polish.
- Small `d3-*` utilities only if needed, such as scales.

### Avoid For V1

- Full D3 unless a specific visualization really needs it.
- Three.js.
- GSAP.
- p5.js.
- Heavy animation frameworks.

### Rendering Medium Guidance

Prefer SVG first for:

- Consistent hashing rings
- DAGs
- Resolver trees
- CDN/cache topologies
- Key movement and assignment animations

Use Canvas only when element counts or frame-rate requirements make SVG awkward.

## Interactive Demo Component Pattern

### Principle

Each demo should be self-contained and co-located with its post, but use a shared shell for consistent controls, spacing, and captions.

### Shared Primitives

Suggested shared components:

- `app/components/interactive/DemoFrame.tsx`
- `app/components/interactive/PlaybackControls.tsx`
- `app/components/interactive/RangeControl.tsx`
- `app/components/interactive/StepControls.tsx`
- `app/components/interactive/usePlayback.ts`
- `app/components/interactive/useReducedMotion.ts`
- `app/components/interactive/useResizeObserver.ts`

### Post-Local Structure

- `content/posts/<slug>/demos/*.tsx`
- `content/posts/<slug>/sim/*.ts`
- `content/posts/<slug>/assets/*`

### Suggested Demo Shell

```tsx
type DemoFrameProps = {
  title: string;
  children: React.ReactNode;
  controls?: React.ReactNode;
  caption?: React.ReactNode;
};

export function DemoFrame({
  title,
  children,
  controls,
  caption,
}: DemoFrameProps) {
  return (
    <figure>
      <div>{children}</div>
      {controls}
      {caption ? <figcaption>{caption}</figcaption> : null}
    </figure>
  );
}
```

### MDX Authoring Pattern

```mdx
import { RingIntroDemo } from "./demos/RingIntroDemo";

## A ring, not a modulo trick

<RingIntroDemo />
```

### Authoring Rules

- Prose stays in MDX.
- Each demo exports one clear component.
- Simulation logic lives with the post.
- Shared controls and frame styling live globally.
- Every demo includes a static fallback image or explanatory still.
- Do not autoplay when `prefers-reduced-motion` is enabled.
- Use native buttons and range inputs first.

### Interaction Primitives To Support

- Play/pause
- Single-step forward/backward
- Speed control
- Sliders
- Hover/tap tooltips
- Responsive scaling

### Accessibility Requirements

- 44px minimum touch targets.
- Keyboard access for controls.
- Space toggles play/pause when focused.
- Arrow keys can step through sequences when relevant.
- Tooltips must not be hover-only; mirror key info in a visible panel on mobile.
- Each demo needs a caption that preserves the lesson without JS.

## First Post Recommendation

Start with: `Consistent Hashing`

Why this is the best first post:

- Strong visual mental model.
- Clear before/after animations.
- Relevant to distributed systems.
- Easier to scope than DAG resolution or full GraphQL execution.
- Good fit for SVG and step-through controls.

### Working Title

`Consistent Hashing, Explained By Watching Keys Move`

## First Post Blueprint

### Section 1: The Problem With Naive Sharding

Prose:

- Introduce `hash(key) % N`.
- Explain why adding/removing a node forces massive reassignment.

Demo:

- Compare modulo-based partitioning before and after adding a node.

Controls:

- Key count
- Node count
- Add/remove node

Learning outcome:

- Naive rebalancing moves most keys.

Complexity:

- Low

### Section 2: The Ring Mental Model

Prose:

- Explain hashing both nodes and keys onto a shared ring.
- Explain clockwise ownership.

Demo:

- Visual ring with nodes and keys placed on the circumference.
- Highlight the node that owns the selected key.

Controls:

- Step
- Play/pause
- Hover/tap a key

Learning outcome:

- Ownership is based on intervals on the ring.

Complexity:

- Medium

### Section 3: What Happens When A Node Joins

Prose:

- Show why only one interval is affected.

Demo:

- Animate a new node joining the ring and only nearby keys moving.

Controls:

- Add node
- Remove node
- Speed
- Step through redistribution

Learning outcome:

- Key movement becomes localized.

Complexity:

- Medium

### Section 4: Why Virtual Nodes Matter

Prose:

- Explain load balance and skew.

Demo:

- Compare the same set of machines with different vnode counts.
- Show a simple distribution histogram or load summary.

Controls:

- Virtual node count: `1 / 4 / 16 / 64`
- Machine count
- Randomize layout

Learning outcome:

- More virtual nodes generally improve distribution smoothness.

Complexity:

- Medium

### Section 5: Real-World Caveats

Mostly prose, optionally one light diagram.

Topics:

- Replication
- Uneven hardware
- Hot keys
- Hash quality
- Operational trade-offs

## Complexity Estimate

For a first weekend-sized version:

- Modulo vs ring demo: low complexity
- Node join redistribution demo: medium complexity
- Virtual nodes comparison: medium complexity

Recommended v1 scope:

- Build only two demos first:
  1. modulo vs ring
  2. node join redistribution

## Repo Context Notes

Relevant current files in the repo:

- `lib/api.tsx` currently handles post loading.
- `app/blog/[slug]/page.tsx` currently renders blog posts with `Remarkable`.
- `app/mumblings/[slug]/page.tsx` and `app/projects/[slug]/page.tsx` use the same approach.
- `app/components/blog/body/PostBody.tsx` currently injects HTML strings.
- `app/rss/route.ts` and the OG image routes depend on frontmatter data from the same loader.

This means the safest initial change is to evolve the loading/rendering path without breaking list pages, RSS, or existing post metadata behavior.
